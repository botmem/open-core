<objective>
Reclassify contact entity types by cross-referencing memory entity data.

Purpose: 99.9% of contacts are typed as 'person' because resolveContact only sets entityType on new contacts. Auto-merge rules (Plan 02) depend on accurate entityType to distinguish organizations, products, locations from persons. This plan fixes the entityType for all existing contacts by looking at entity classifications in their linked memories.

Output: `reclassifyEntityTypes()` method on ContactsService + `POST /people/reclassify` endpoint. Also updates `resolveContact()` to set entityType on existing contacts when called with a non-person type.
</objective>

  <action>
Add a `reclassifyEntityTypes()` method to `ContactsService` that:

1. Queries all contacts where entityType is 'person' (or NULL/empty, coalesced to 'person')
2. For each person-typed contact, JOINs through memoryContacts to memories to read the entities JSON array
3. For each linked memory, parses `entities` JSON and finds entries where `entity.value` case-insensitively matches the contact's displayName
4. Counts occurrences of each non-person canonical type (organization, product, location, event, topic, pet, group, device)
5. If ANY non-person type has 1+ occurrences, reclassifies the contact to the most-common non-person type
6. Updates the contact's entityType and updatedAt

NON_PERSON_TYPES to check: `['organization', 'product', 'location', 'event', 'topic', 'pet', 'group', 'device']`

Performance note: There are ~1200 contacts. For each, query linked memories. This is a one-time operation, so O(N) queries are acceptable. Do NOT try to load all memories into memory at once.

Return shape: `{ reclassified: number, details: Array<{ contactId: string, displayName: string, oldType: string, newType: string }> }`

Also add `POST /people/reclassify` endpoint in the controller that calls `reclassifyEntityTypes()` and returns the result.
