# Phase 8.1: Contact Auto-Merge - Research

**Researched:** 2026-03-08
**Domain:** Contact deduplication / auto-merge logic in SQLite + Drizzle ORM
**Confidence:** HIGH

## Summary

Phase 8.1 adds automatic merging of obvious contact duplicates to eliminate manual review for safe cases. The existing codebase has a mature `mergeContacts()` method (transaction-safe, handles identifiers, memory links, avatars, dismissals) and a `getSuggestions()` method that identifies merge candidates. The auto-merge feature hooks into the suggestion pipeline, automatically executing merges for high-confidence pairs instead of surfacing them for manual review.

**Critical prerequisite gap:** The current database has 1199 contacts, of which 1198 are typed as `person` and 1 as `device`. Zero contacts have entity types like `organization`, `product`, `location`, etc. The Phase 8.1 success criteria require entityType-based auto-merge rules (e.g., "both contacts have entityType in organization/product/location/event/topic"). This means Phase 8.1 MUST include a contact entityType reclassification step -- either by cross-referencing entity data from associated memories or by running a lightweight heuristic on the contact's name and identifiers.

**Primary recommendation:** Build auto-merge as a method on `ContactsService` that runs in two waves: (1) reclassify contact entity types from memory entity data, then (2) apply auto-merge rules to exact-name pairs. Expose via a `POST /people/auto-merge` endpoint and optionally integrate into `getSuggestions()` to auto-merge before returning remaining suggestions.

## Architecture Patterns

### Recommended Approach

The auto-merge system has two distinct concerns:

**Concern 1: Contact EntityType Reclassification**
- Most contacts were created before canonical entity types existed in the enrichment pipeline
- `resolveContact()` only sets entityType on NEW contacts; existing contacts keep their original type
- The memory entities table contains type classifications (person, organization, location, etc.) that can be used to reclassify contacts
- Approach: For each contact, look at entity types in linked memories. If a contact is consistently identified as "organization" across its memories, reclassify it

**Concern 2: Auto-Merge Rules**
Three tiers of exact-name matches with different safety profiles:

| Tier | Condition | Action | Safety |
|------|-----------|--------|--------|
| 1 - Non-person entities | Both contacts have entityType in (organization, product, location, event, topic) AND exact name match | Auto-merge | HIGH - organizations with the same name are almost certainly the same entity |
| 2 - Sparse-to-rich | One contact has ONLY name identifiers, other has structured identifiers (email/phone/slack_id) AND exact name match | Auto-merge sparse into rich | HIGH - sparse contacts are incomplete duplicates from a different connector |
| 3 - Person-to-person | Both are person type with different structured identifiers | Keep as manual suggestion | N/A - ambiguous, needs human judgment |

### Pattern 1: Contact EntityType Reclassification via Memory Entities

**What:** Cross-reference contacts with their linked memories' entity arrays to determine the correct entityType.
**When to use:** One-time reclassification + ongoing during auto-merge.

```typescript
// Query: For each contact, find the most common entity type in associated memories
// JOIN memoryContacts -> memories, parse entities JSON, aggregate type counts
async reclassifyEntityTypes(): Promise<{ reclassified: number }> {
  const db = this.dbService.db;

  // Get all person-type contacts (candidates for reclassification)
  const personContacts = await db.select().from(contacts)
    .where(eq(contacts.entityType, 'person'));

  let reclassified = 0;
  const NON_PERSON_TYPES = new Set(['organization', 'product', 'location', 'event', 'topic']);

  for (const contact of personContacts) {
    // Find entity types from linked memories
    const linkedMemories = await db.select({ entities: memories.entities })
      .from(memoryContacts)
      .innerJoin(memories, eq(memoryContacts.memoryId, memories.id))
      .where(eq(memoryContacts.contactId, contact.id));

    // Count entity type occurrences for this contact's display name
    const typeCounts = new Map<string, number>();
    for (const mem of linkedMemories) {
      const entities = JSON.parse(mem.entities || '[]');
      for (const e of entities) {
        if (e.value?.toLowerCase() === contact.displayName.toLowerCase() && NON_PERSON_TYPES.has(e.type)) {
          typeCounts.set(e.type, (typeCounts.get(e.type) || 0) + 1);
        }
      }
    }

    // If majority of mentions are a non-person type, reclassify
    if (typeCounts.size > 0) {
      const [bestType] = [...typeCounts.entries()].sort((a, b) => b[1] - a[1])[0];
      await db.update(contacts)
        .set({ entityType: bestType, updatedAt: new Date().toISOString() })
        .where(eq(contacts.id, contact.id));
      reclassified++;
    }
  }

  return { reclassified };
}
```

### Pattern 2: Auto-Merge with Safety Tiers

**What:** Iterate exact-name pairs, apply tier rules, merge safe pairs automatically.
**When to use:** Called before `getSuggestions()` returns, or via explicit endpoint.

```typescript
async autoMerge(): Promise<{ merged: number; byRule: Record<string, number> }> {
  const db = this.dbService.db;
  let merged = 0;
  const byRule: Record<string, number> = { nonPerson: 0, sparseToRich: 0 };

  // Group contacts by lowercase display name
  const allContacts = await db.select().from(contacts);
  const byName = new Map<string, typeof allContacts>();
  for (const c of allContacts) {
    const key = c.displayName.toLowerCase().trim();
    if (!byName.has(key)) byName.set(key, []);
    byName.get(key)!.push(c);
  }

  for (const [, group] of byName) {
    if (group.length < 2) continue;

    // Rule 1: Non-person entities with same name -> auto-merge
    const NON_PERSON = new Set(['organization', 'product', 'location', 'event', 'topic']);
    const nonPersonGroup = group.filter(c => NON_PERSON.has(c.entityType));
    if (nonPersonGroup.length >= 2) {
      const target = nonPersonGroup[0];
      for (const source of nonPersonGroup.slice(1)) {
        await this.mergeContacts(target.id, source.id);
        byRule.nonPerson++;
        merged++;
      }
    }

    // Rule 2: Sparse-to-rich exact name -> auto-merge sparse into rich
    // (handled per-pair within remaining contacts)
  }

  return { merged, byRule };
}
```

### Anti-Patterns to Avoid
- **Fuzzy matching for auto-merge:** Only use EXACT name matches for automatic merges. Fuzzy/substring matches should remain as manual suggestions -- false positive merges are destructive and hard to undo.
- **Merging person-to-person without manual review:** Two people can have the same name. Never auto-merge person-type contacts with different structured identifiers.
- **Running auto-merge outside a transaction:** Each individual merge already uses a transaction, but the auto-merge loop should handle the case where a contact was already merged by a previous iteration (the existing `mergeContacts` handles this -- returns early if source is already deleted).

## Common Pitfalls

### Pitfall 1: Entity Type Data Not Available on Contacts
**What goes wrong:** Auto-merge rules check `contacts.entityType` but 99.9% of contacts are typed as 'person' regardless of actual type.
**Why it happens:** `resolveContact()` only sets entityType on new contacts. All 1198 existing contacts were created before the canonical entity type system.
**How to avoid:** Phase 8.1 MUST include a reclassification step that infers entity type from memory entities before applying auto-merge rules.
**Warning signs:** Auto-merge reports 0 non-person matches.

### Pitfall 2: Phase 8 Migration Not Yet Run
**What goes wrong:** The entity backfill migration (`backfill-entity-types.ts`) exists but the database still has non-canonical entity types (time, amount, metric, etc.) in memory entities.
**Why it happens:** Phase 8 plans were created but the migration hasn't been executed against the actual database.
**How to avoid:** Verify or run the Phase 8 migration before attempting contact reclassification. The reclassification depends on canonical entity types in memory entities.
**Warning signs:** Memory entity types include "time", "amount", "transaction_id", etc.

### Pitfall 3: Merge Cascades
**What goes wrong:** Merging A into B changes B's identifiers, which might now match C, triggering another merge.
**Why it happens:** `resolveContact()` already has auto-merge for shared non-name identifiers. The new auto-merge by name could cascade.
**How to avoid:** Build the auto-merge as a batch operation that groups by name FIRST, then merges all members of each group into one target. Don't iterate pair-by-pair with re-scanning.
**Warning signs:** Auto-merge takes unexpectedly long or reports more merges than expected.

### Pitfall 4: Non-Idempotent Merge
**What goes wrong:** Running auto-merge twice causes errors or duplicates.
**Why it happens:** After merge, source contact is deleted. If not checked, second run tries to merge a deleted contact.
**How to avoid:** `mergeContacts()` already handles this -- it returns early if either side doesn't exist. The auto-merge loop should re-query contacts at the start (not reuse stale data) and skip contacts that no longer exist.
**Warning signs:** Errors about missing contacts on second run.

### Sparse Contact Detection

A contact is "sparse" when it has only `name` type identifiers and no structured identifiers (email, phone, slack_id, etc.):

```sql
SELECT c.id FROM contacts c
WHERE NOT EXISTS (
  SELECT 1 FROM contact_identifiers ci
  WHERE ci.contact_id = c.id
  AND ci.identifier_type NOT IN ('name')
)
```

Current count: 0 sparse contacts. But this may change after reclassification or future syncs.

## Open Questions

1. **Should auto-merge run automatically during getSuggestions() or only on explicit trigger?**
   - What we know: getSuggestions() is called when the user views the merge queue in the UI
   - What's unclear: Whether the user expects auto-merge to happen proactively (on sync) or reactively (when viewing suggestions)
   - Recommendation: Run during getSuggestions() so the user sees a clean queue, but also expose a manual endpoint for CLI/testing

2. **Should resolveContact() be updated to set entityType on existing contacts?**
   - What we know: Currently only sets entityType on NEW contacts
   - What's unclear: Whether updating entityType on existing contacts during resolve could cause issues
   - Recommendation: Yes -- if a contact exists and resolveContact is called with a non-person entityType, update it. This prevents re-accumulating mistyped contacts after the reclassification.

3. **How to handle contacts with 3+ duplicates in a group?**
   - What we know: Some names like "Apple" have 2 duplicate contacts, but after reclassification there could be more
   - What's unclear: Whether to merge all into one or just pairs
   - Recommendation: Merge all members of a name-group into the first (richest) contact in one pass
