**Migration pattern for services with a `private get db()` shortcut (accounts.service.ts, jobs.service.ts):**

Before:

```typescript
private get db() { return this.dbService.db; }

async getById(id: string) {
  const [row] = await this.db.select().from(accounts).where(eq(accounts.id, id));
  return row ?? null;
}
```

After (remove the private getter; wrap each method):

```typescript
async getById(id: string) {
  return this.dbService.withCurrentUser(async (db) => {
    const [row] = await db.select().from(accounts).where(eq(accounts.id, id));
    return row ?? null;
  });
}
```

**Migration pattern for services using `this.dbService.db` directly (contacts.service.ts, memory.service.ts):**

Before:

```typescript
const results = await this.dbService.db.select().from(memories).where(...);
```

After:

```typescript
const results = await this.dbService.withCurrentUser((db) =>
  db.select().from(memories).where(...)
);
```

**Scope:** Only wrap methods that query RLS-protected tables (memories, accounts, contacts, jobs, raw_events, memory_banks, api_keys, memory_links, memory_contacts, contact_identifiers, refresh_tokens, password_resets). Methods that only query `users`, `settings`, or `connector_credentials` do not need wrapping.

**Return type handling:** Methods that currently return `Promise<T>` keep the same signature — `withCurrentUser()` is transparent in return type.

**Transaction note:** Each `withCurrentUser()` call is its own transaction. If a service method performs multiple logically-related writes, wrap all of them in a single `withCurrentUser()` call to keep them in one transaction.

BullMQ processors run outside the HTTP request lifecycle — RlsInterceptor does not fire for them. They must use `withUserId(ownerUserId, fn)` explicitly.

**How processors resolve ownerUserId:**

In embed.processor.ts: `ownerUserId` is already resolved early in `process()` by querying accounts:

```typescript
const acct = await this.dbService.db
  .select({ userId: accounts.userId })
  .from(accounts)
  .where(eq(accounts.id, rawEvent.accountId));
ownerUserId = acct[0]?.userId || null;
```

This accounts lookup itself queries an RLS-protected table. Bootstrap it outside the withUserId wrapper by using the raw pool before RLS is active, or use a service account role. The safest approach for this lookup: use `this.dbService.withUserId` is not yet available without userId. Instead, read from the `users` table (not RLS-protected) via a JOIN, or look up account by its ID using a direct pool query bypassing RLS for this single bootstrap read.

Practical approach: Perform the initial account→userId lookup using `this.dbService.db` directly (unscoped — this query runs as the DB app user which must have SELECT on accounts without RLS restriction, or add a BYPASSRLS policy exception for the app user on the accounts lookup). Then wrap all subsequent queries in `withUserId(ownerUserId)`.

**Migration pattern for processors:**

Before:

```typescript
async process(job: Job) {
  const { rawEventId } = job.data;
  const rawEvent = await this.dbService.db.select().from(rawEvents).where(eq(rawEvents.id, rawEventId));
  // ... lots of queries using this.dbService.db
}
```

After:

```typescript
async process(job: Job) {
  const { rawEventId } = job.data;
  // Step 1: Bootstrap — resolve ownerUserId from account (unscoped lookup)
  const rawEvent = await this.dbService.db.select().from(rawEvents).where(eq(rawEvents.id, rawEventId));
  const acct = await this.dbService.db.select({ userId: accounts.userId }).from(accounts).where(eq(accounts.id, rawEvent[0].accountId));
  const ownerUserId = acct[0]?.userId;
  if (!ownerUserId) throw new Error(`No ownerUserId for rawEvent ${rawEventId}`);

  // Step 2: All remaining queries within withUserId scope
  await this.dbService.withUserId(ownerUserId, async (db) => {
    // use db.select(), db.insert(), db.update(), db.delete() here
    // replace all this.dbService.db.xxx calls with db.xxx
  });
}
```

Apply this pattern to:

- embed.processor.ts: wrap the main body of process() after ownerUserId is resolved
- enrich.processor.ts: resolve ownerUserId from the memory→account join, then wrap the rest
- backfill.processor.ts: resolve ownerUserId from the memory's accountId, wrap the rest
- sync.processor.ts: sync operates on account-level, resolve userId from account, wrap DB writes

**Note on services called from processors:** Processors call MemoryService, AccountsService, ContactsService methods. Those service methods now use `withCurrentUser()` — which falls back to unscoped `fn(this.db)` when no RlsContext is set (BullMQ context). This means processor-invoked service calls bypass RLS unless the service is updated to also accept an explicit `db` parameter. For now, service methods called from processors must be updated to accept an optional `db?: NodePgDatabase<typeof schema>` parameter and use it when provided, or processors must inline their queries rather than delegating to services. Use whichever approach is cleaner per service method. Document the choice inline.

**RLS enforcement verification (run after all processor changes compile):**

After the API restarts with the new code, prove RLS is working with a direct psql check:

```bash
# Get two real user IDs from the database (requires at least 2 users to exist)
# If only 1 user exists, create a second test user via POST /api/auth/register
psql $DATABASE_URL -c "
  DO \$\$
  DECLARE
    uid_a text;
    uid_b text;
    mem_count int;
  BEGIN
    SELECT id INTO uid_a FROM users LIMIT 1 OFFSET 0;
    SELECT id INTO uid_b FROM users LIMIT 1 OFFSET 1;

    IF uid_b IS NULL THEN
      RAISE NOTICE 'Only one user exists — skipping cross-user RLS check';
      RETURN;
    END IF;

    -- Set session as user B, count memories belonging to user A's accounts
    PERFORM set_config('app.current_user_id', uid_b, true);
    SELECT COUNT(*) INTO mem_count
      FROM memories
      WHERE account_id IN (SELECT id FROM accounts WHERE user_id = uid_a);

    IF mem_count > 0 THEN
      RAISE EXCEPTION 'RLS BREACH: user B can see % memories belonging to user A', mem_count;
    ELSE
      RAISE NOTICE 'RLS OK: cross-user query returned 0 rows';
    END IF;
  END
  \$\$;
"
```

Expected output: `NOTICE: RLS OK: cross-user query returned 0 rows`
If output is `ERROR: RLS BREACH`: the policies from Plan 01 are misconfigured or the DB role has BYPASSRLS — investigate and fix before declaring this plan complete.
