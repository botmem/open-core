  <action>
    **1. Extend SearchFilters interface** in `memory.service.ts`:
    Add `from?: string` and `to?: string` (ISO 8601 datetime strings) to the existing `SearchFilters` interface.

    **2. Extend SearchResponse interface** in `memory.service.ts`:
    Add `parsed?: ParsedQuery` to `SearchResponse`. Define `ParsedQuery`:
    ```typescript
    export interface ParsedQuery {
      temporal: { from: string; to: string } | null;
      temporalFallback?: boolean;
      entities: { id: string; displayName: string }[];
      intent: 'recall' | 'browse' | 'find';
      cleanQuery: string;
    }
    ```

    **3. Wire NLQ parser into `search()` method** at `memory.service.ts:176`:
    At the top of `search()`, after the empty-query check:
    - `import { parseNlq } from './nlq-parser';`
    - Call `const parsed = parseNlq(query);`
    - Apply temporal filters: if `parsed.temporal`, set `effectiveFilters.from = parsed.temporal.from` and `.to = parsed.temporal.to` (only if caller didn't already provide from/to)
    - Apply sourceType hint: if `parsed.sourceTypeHint && !filters?.sourceType`, set `effectiveFilters.sourceType = parsed.sourceTypeHint`
    - Apply intent-based limit: if `parsed.intent === 'find'`, cap effective limit to 5; if `parsed.intent === 'browse'`, keep caller limit
    - Use `parsed.cleanQuery` instead of raw `query` for the embedding call (`this.ollama.embed(parsed.cleanQuery)`)
    - Keep using original `query` for entity resolution (entity names should not be stripped)

    **4. Implement browse intent weight adjustment** in `computeWeights()`:
    Add an optional `intent` parameter to `computeWeights()`. When `intent === 'browse'`:
    - Without reranker: `0.40 * semantic + 0.40 * recency + 0.15 * importance + 0.05 * trust` (instead of 0.70/0.15/0.10/0.05)
    - With reranker: `0.25 * semantic + 0.20 * rerank + 0.40 * recency + 0.10 * importance + 0.05 * trust`

    **5. Implement temporal fallback:**
    After the main search returns results, if `parsed.temporal` was applied and `results.length === 0`:
    - Re-run search WITHOUT temporal filters (remove from/to from effectiveFilters)
    - Set `temporalFallback: true` in the parsed response field

    **6. Build the `parsed` response field:**
    Include in every SearchResponse: `parsed: { temporal: parsed.temporal, temporalFallback, entities: resolvedContacts.map(c => ({ id: c.id, displayName: c.displayName })), intent: parsed.intent, cleanQuery: parsed.cleanQuery }`

    **7. Extend `buildQdrantFilter()`** at `memory.service.ts:1090`:
    Add temporal range condition:
    ```typescript
    if (filters.from || filters.to) {
      const range: Record<string, string> = {};
      if (filters.from) range.gte = filters.from;
      if (filters.to) range.lte = filters.to;
      must.push({ key: 'event_time', range });
    }
    ```

    **8. Add Qdrant datetime payload index** in `qdrant.service.ts`:
    In `onModuleInit()`, after `ensureCollection` and `ensureIndexed`, add:
    ```typescript
    try {
      await this.client.createPayloadIndex(QdrantService.COLLECTION, {
        field_name: 'event_time',
        field_schema: 'datetime',
      });
    } catch (err: any) {
      // 400 = index already exists, safe to ignore
      if (err?.status !== 400 && !err?.message?.includes('already exists')) {
        console.error('Failed to create event_time payload index:', err);
      }
    }
    ```

    **Important:** Pass `effectiveFilters` (with temporal) to both `buildQdrantFilter()` and the SQL text search WHERE clauses. For SQL, add `AND eventTime >= ? AND eventTime <= ?` conditions when from/to are present.
