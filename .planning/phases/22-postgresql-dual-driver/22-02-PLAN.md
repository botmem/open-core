**memory.service.ts -- FTS rewrite:**
Find the FTS5 query block (around line 381-387) where `this.dbService.sqlite.prepare('SELECT id FROM memories_fts WHERE memories_fts MATCH ? LIMIT ?')` is used. Replace with tsvector + pg_trgm search using Drizzle `sql` template:

```typescript
// tsvector search: each word as prefix, AND logic
const ftsQuery = searchWords.map(w => `${w}:*`).join(' & ');
const ftsResults = await this.dbService.db
  .select({ id: memories.id })
  .from(memories)
  .where(sql`(to_tsvector('english', ${memories.text}) || to_tsvector('simple', ${memories.text})) @@ to_tsquery('simple', ${ftsQuery})`)
  .limit(limit * 2);
for (const r of ftsResults) textMatchIds.add(r.id);
```

IMPORTANT: Since memories.text is encrypted at rest, the tsvector index won't work on encrypted text. The FTS query should still be attempted but will only match unencrypted text. This is a known limitation documented in RESEARCH.md (encrypted fields can't be indexed). For now, keep the query as-is -- it will gracefully return empty results for encrypted memories, and vector search (Qdrant) remains the primary search path.

Also ensure the search function is fully async (it should be already, but verify all DB calls use await).

This task applies three categories of mechanical fixes across the codebase, then rewrites test infrastructure.

**Category 1: Remove JSON.parse on JSONB columns**

Search all files for `JSON.parse` calls on columns that are now jsonb: `factuality`, `weights`, `avatars` (contacts), `metadata` (contacts only -- memories.metadata is still text/encrypted).

For each occurrence:
- If it reads `JSON.parse(row.factuality)` -> just use `row.factuality` directly (already an object)
- If it reads `JSON.parse(row.weights)` -> just use `row.weights` directly
- If it reads `JSON.parse(row.avatars)` -> just use `row.avatars` directly
- If it reads `JSON.parse(row.metadata)` on contacts -> just use `row.metadata` directly

DO NOT change JSON.parse on these columns (they stay as text because they're encrypted):
- memories.text (encrypted)
- memories.entities (encrypted)
- memories.claims (encrypted)
- memories.metadata (encrypted -- note: contacts.metadata IS jsonb, memories.metadata is NOT)
- accounts.authContext (encrypted)
- connectorCredentials.credentials (encrypted)

Also check for `JSON.stringify` calls when writing to jsonb columns -- jsonb accepts objects directly, no stringification needed. But for encrypted text columns, JSON.stringify is still needed before encryption.

**Category 2: Boolean 0/1 -> true/false**

Search for patterns:
- `=== 0`, `=== 1`, `== 0`, `== 1` on boolean fields (pinned, onboarded, isDefault)
- `.default(0)` or `.default(1)` in Drizzle insert values
- `pinned: 0`, `pinned: 1`, `onboarded: 0`, `onboarded: 1`, `isDefault: 0`, `isDefault: 1`
- `eq(field, 0)`, `eq(field, 1)` in Drizzle where clauses

Replace with `true`/`false` equivalents.

Key files:
- memory-banks.service.ts: `isDefault: 1` -> `isDefault: true`, `eq(memoryBanks.isDefault, 1)` -> `eq(memoryBanks.isDefault, true)`
- memory.service.ts: `pinned: 1`/`pinned: 0` -> `pinned: true`/`pinned: false`
- Any user onboarding code: `onboarded: 0`/`onboarded: 1` -> `onboarded: false`/`onboarded: true`

**Category 3: Timestamp handling**

Timestamp columns now return Date objects instead of ISO strings. Check all services for:
- `new Date().toISOString()` when writing timestamps -> change to `new Date()` (Drizzle handles Date->timestamptz)
- `new Date(row.createdAt)` when reading timestamps -> just use `row.createdAt` (already a Date)
- String comparisons on timestamp fields -> use Date comparisons
- Any `.toISOString()` calls on read values that are already Dates -> remove redundant conversion

For API responses that return timestamps to the frontend, Drizzle will return Date objects which JSON.stringify serializes as ISO strings automatically. No changes needed in controllers/responses.
